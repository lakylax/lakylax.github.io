Project Overview

I designed and implemented a fully normalized relational database system from scratch as a semester-long project, applying core database engineering principles including entity-relationship modeling, normalization theory, and advanced SQL programming.

The system manages academic data — including authors, theses, departments, and grading — using Microsoft SQL Server accessed through Azure Data Studio. I built the complete data pipeline from conceptual design through physical implementation, incorporating stored procedures, transactions with savepoints, views, and triggers to ensure data integrity and operational efficiency.

🏗️

Database Design

Entity-Relationship Model

I designed the conceptual schema using an Entity-Relationship diagram to map out all entities, their attributes, and the relationships between them. The ER model captures the academic domain with entities for Authors, Theses, and Departments, establishing clear cardinality constraints and participation requirements.

Entity-Relationship Diagram — Conceptual database schema showing entities and relationships

The design captures key relationships: each author belongs to a department, authors produce theses, and departments house multiple academic programs. I used Azure Data Studio as my primary development environment for implementing and querying the database.

Normalization Process

From Unnormalized Data to 3NF

I applied systematic normalization to eliminate data redundancy and ensure referential integrity. Starting from raw unnormalized data, I progressively decomposed relations through each normal form.

Initial Dataset (Unnormalized)

Raw dataset before normalization — containing repeating groups and redundancies
1NF

First Normal Form

  • Eliminated repeating groups from all tables
  • Ensured atomic values in every column
  • Established primary keys for unique row identification
2NF

Second Normal Form

  • Removed partial dependencies on composite keys
  • Each non-key attribute fully depends on the entire primary key
  • Separated department data into its own relation
3NF

Third Normal Form

  • Eliminated transitive dependencies
  • All non-key attributes depend only on the primary key
  • Database achieved full 3NF compliance

After 1NF Decomposition

Data structure after applying First Normal Form — atomic values, primary keys established

Fully Normalized Schema (2NF/3NF)

Implementation

Stored Procedures

I implemented stored procedures to encapsulate frequently-used business logic, improving performance and maintaining a clean separation between application code and data access patterns.

CREATE PROCEDURE [dbo].[Get10top] AS BEGIN SELECT TOP 10 last_name, first_name FROM dbo.Author AS a ORDER BY Average_Grade_From_Studies DESC END;

This procedure retrieves the top 10 highest-performing authors ranked by their academic grades — a common query pattern I optimized by encapsulating it as a reusable procedure.

CREATE PROCEDURE [dbo].[Get_Email_Address] AS BEGIN SELECT Email_address FROM dbo.Author AS a END;

I designed this procedure to provide a quick lookup of all author contact information — useful for batch operations like sending notifications or generating mailing lists.

Transaction Management & ACID Compliance

I implemented transactions with explicit control flow to ensure data integrity during multi-step operations. Each transaction demonstrates proper use of COMMIT, ROLLBACK, and SAVEPOINT mechanisms.

A

Atomicity

All-or-nothing execution of operations

C

Consistency

Database remains valid after each transaction

I

Isolation

Concurrent transactions don't interfere

D

Durability

Committed changes persist through failures

Transaction state diagram — Active → Partially Committed → Committed / Failed → Aborted

Transaction with COMMIT

BEGIN TRANSACTION DELETE FROM Department WHERE department_Id = 0 UPDATE Department SET department_name = 'Faculty of History' WHERE department_name = 'Faculty of Classics' COMMIT SELECT * FROM Department

I used this transaction to perform a data cleanup operation — removing invalid records and correcting department names in a single atomic unit. Both operations either succeed together or fail together.

Transaction with SAVEPOINT & ROLLBACK

BEGIN TRANSACTION UPDATE Thesis SET Final_grade = '4' WHERE Final_grade = '3' SAVE TRANSACTION A -- Savepoint established DELETE Thesis WHERE Thesis_Id = 4 ROLLBACK TRANSACTION A -- Undo delete, keep grade update COMMIT SELECT * FROM Thesis

This demonstrates my implementation of partial rollback using savepoints — the grade update persists while the deletion is undone. This pattern is essential for complex operations where only specific steps may need reversal.

Database Views

I created views to provide simplified, secure access to specific data subsets — abstracting complex table structures into clean virtual tables for application consumption.

CREATE VIEW Only_Thesis_Titles AS SELECT Thesis_title FROM dbo.Thesis;

This view exposes only thesis titles — restricting access to sensitive grading and author information while providing a clean interface for title-only lookups.

Database Triggers

I implemented triggers to automate referential actions — when new author records are inserted, related records in dependent tables (taxes, vacations, salaries) are automatically created, maintaining data consistency without requiring application-level logic.

Key Takeaways

Design Before Implementation

Starting with a solid ER model and normalizing before writing any SQL saved significant refactoring time and produced a cleaner schema.

Normalization Eliminates Anomalies

Achieving 3NF removed update, insertion, and deletion anomalies — the database maintains consistency without application-level guards.

Transactions Protect Integrity

Using savepoints and explicit commit/rollback patterns ensures complex multi-step operations remain atomic even when partial failures occur.

Stored Procedures Centralize Logic

Encapsulating business logic in the database layer improves performance and ensures consistent behavior across all consuming applications.

Views Enforce Security

Virtual tables provide controlled data exposure — restricting what applications can see without duplicating data or adding middleware.

Triggers Automate Consistency

Event-driven automation ensures referential integrity is maintained even when applications bypass normal data entry paths.
